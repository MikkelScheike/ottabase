# RBAC and Audit Logging Implementation

This document describes the RBAC (Role-Based Access Control) and audit logging implementation for the Ottabase monorepo.

## Overview

The implementation consists of:

1. **Database Models** - Role, Permission, UserRole, and AuditLog models in @ottabase/ottaorm
2. **RBAC Package** - `@ottabase/rbac` for role and permission-based access control
3. **Audit Package** - `@ottabase/audit` for audit logging utilities and middleware
4. **Logger Integration** - Database transport for @ottabase/logger to persist audit logs

## Architecture

### Database Schema

#### Roles Table

```sql
CREATE TABLE roles (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL UNIQUE,
  description TEXT,
  permissions TEXT NOT NULL, -- JSON array of permission names
  is_system BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);
```

#### Permissions Table

```sql
CREATE TABLE permissions (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL UNIQUE,
  description TEXT,
  resource TEXT NOT NULL,
  action TEXT NOT NULL,
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);
```

#### User_Roles Table (Junction)

```sql
CREATE TABLE user_roles (
  user_id TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  role_id TEXT NOT NULL REFERENCES roles(id) ON DELETE CASCADE,
  organization_id TEXT, -- Optional for multi-tenant
  assigned_at TIMESTAMP,
  assigned_by TEXT,
  PRIMARY KEY (user_id, role_id)
);
```

#### Audit_Logs Table

```sql
CREATE TABLE audit_logs (
  id TEXT PRIMARY KEY,
  user_id TEXT REFERENCES users(id) ON DELETE SET NULL,
  user_email TEXT,
  action TEXT NOT NULL,
  resource_type TEXT NOT NULL,
  resource_id TEXT,
  changes TEXT, -- JSON before/after values
  metadata TEXT, -- JSON additional context
  ip_address TEXT,
  user_agent TEXT,
  status TEXT DEFAULT 'success', -- success, failure, error
  error_message TEXT,
  created_at TIMESTAMP
);
```

## Package Structure

### @ottabase/ottaorm Models

Located in `/packages/ottaorm/src/models/`:

- `Role.ts` / `Role.schema.ts` - Role model with permission management
- `Permission.ts` / `Permission.schema.ts` - Permission model with wildcard matching
- `UserRole.ts` / `UserRole.schema.ts` - Junction table for user-role relationships
- `AuditLog.ts` / `AuditLog.schema.ts` - Audit log model with querying utilities

Extended `User.ts` with RBAC methods:

- `assignRole()`, `removeRole()`
- `hasRole()`, `hasAnyRole()`, `hasAllRoles()`
- `hasPermission()`, `hasAnyPermission()`, `hasAllPermissions()`
- `getPermissions()`, `isAdmin()`

## Usage Examples

### 1. Setup (Run Once)

```typescript
import { Role, Permission } from '@ottabase/ottaorm/models';

// Create default roles (admin, editor, viewer)
await Role.ensureDefaultRoles();

// Seed default permissions
await Permission.seedDefaultPermissions();
```

### 2. Assign Roles to Users

```typescript
import { User, Role } from '@ottabase/ottaorm/models';

const user = await User.find('user-id');
const adminRole = await Role.findByName('admin');

// Assign role
await user.assignRole(adminRole.get('id'), 'assigned-by-user-id');

// Check role
const isAdmin = await user.hasRole('admin');

// Check permission
const canCreateUsers = await user.hasPermission('users:create');
```

### 3. Protect API Routes with RBAC

```typescript
// app/api/users/route.ts
import { withRBAC } from '@ottabase/rbac/middleware';
import { User } from '@ottabase/ottaorm/models';

export const GET = withRBAC(
    async (request) => {
        const users = await User.all();
        return Response.json({ users });
    },
    {
        permissions: ['users:read'],
        getUserFromRequest: async (request) => {
            const userId = request.headers.get('x-user-id');
            return userId ? await User.find(userId) : null;
        },
    },
);

export const POST = withRBAC(
    async (request) => {
        const body = await request.json();
        const user = await User.create(body);
        return Response.json({ user });
    },
    {
        roles: ['admin'],
    },
);
```

### 4. Audit Logging

#### Automatic (Middleware)

```typescript
import { withAudit } from '@ottabase/audit/middleware';

export const POST = withAudit(
    async (request) => {
        const body = await request.json();
        const user = await User.create(body);
        return Response.json({ user });
    },
    {
        resourceType: 'user',
        action: 'create',
        includeRequestBody: true,
    },
);
```

#### Manual

```typescript
import { logCreate, logUpdate, extractRequestContext } from '@ottabase/audit/utils';

// Log creation
const context = extractRequestContext(request, userId, userEmail);
await logCreate('user', user.id, userData, context);

// Log update with changes
await logUpdate(
    'user',
    userId,
    {
        name: { from: 'Old', to: 'New' },
        email: { from: 'old@email.com', to: 'new@email.com' },
    },
    context,
);
```

### 5. Combined RBAC + Audit

```typescript
import { withRBAC } from '@ottabase/rbac/middleware';
import { logCreate, extractRequestContext } from '@ottabase/audit/utils';

export const POST = withRBAC(
    async (request) => {
        const body = await request.json();
        const user = await User.create(body);

        // Manual audit logging
        const userId = request.headers.get('x-user-id');
        const context = extractRequestContext(request, userId);
        await logCreate('user', user.get('id'), body, context);

        return Response.json({ user });
    },
    {
        permissions: ['users:create'],
    },
);
```

### 6. Query Audit Logs

```typescript
import { AuditLog } from '@ottabase/ottaorm/models';

// Get all logs for a user
const userLogs = await AuditLog.getByUser('user-id');

// Get logs for a resource
const resourceLogs = await AuditLog.getByResource('user', 'user-id');

// Get recent logs
const recentLogs = await AuditLog.getRecent(100);

// Get failed actions
const failures = await AuditLog.getFailures();

// Get logs in date range
const logs = await AuditLog.getByDateRange(Date.UTC(2024, 0, 1), Date.UTC(2024, 11, 31));
```

## Permission Format

Permissions follow the format `resource:action` with wildcard support:

| Permission     | Description             |
| -------------- | ----------------------- |
| `users:read`   | Read users only         |
| `users:create` | Create users only       |
| `users:*`      | All actions on users    |
| `*:read`       | Read all resources      |
| `*:*`          | All permissions (admin) |

## Default Roles

| Role     | Permissions                      | Description         |
| -------- | -------------------------------- | ------------------- |
| `admin`  | `*:*`                            | Full system access  |
| `editor` | `*:read`, `*:create`, `*:update` | Can create and edit |
| `viewer` | `*:read`                         | Read-only access    |

## Security Considerations

1. **Default deny** - Routes without RBAC middleware should still require authentication
2. **Least privilege** - Assign minimum required permissions
3. **Sensitive data** - Always sanitize before logging
4. **Audit log retention** - Implement retention policies
5. **Permission validation** - Validate permission format before creation

## Performance Considerations

1. **Caching** - Cache user roles and permissions
2. **Buffering** - Use buffered transport for high-volume logging
3. **Indexing** - Add indexes on audit_logs(user_id, resource_type, created_at)
4. **Pagination** - Paginate audit log queries
